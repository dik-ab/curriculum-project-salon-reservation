# Issue Workflow — issueからPRまでの型

各issueは次の7ステップで進めます。ステップを飛ばさないでください(特に2と3)。

## 1. issueを読解する

- `gh issue view <番号>` でタスク・受入条件・ヒントを読む。
- 受入条件を「どうやって検証するか」まで言語化する(手動確認かテストか)。

## 2. 仕様を特定する

- issueの「仕様参照」に挙がっているdocsのセクションを実際に開いて読む。
- 関係するテーブル定義(`docs/database.md`)とエンドポイント定義(`docs/api.md`)を手元に控える。
- 仕様に書かれていない判断が必要になったら、**非同期で進めます**。実装詳細レベルの軽微な判断(メッセージ文言、並び順の細部、UIの細かい挙動など)は自分で妥当な案を決め、**判断と理由をissueコメントに記録してそのまま進んで構いません**。受講者(人間)の回答を待って手を止めるのは、ドメインルールの解釈が割れる場合(金額・予約の成立条件・権限など、どちらに転んでも仕様の意味が変わる判断)のみです。

## 3. 実装計画をissueコメントに書く

`gh issue comment <番号> --body "..."` で、着手前に計画を残します。フォーマット:

```markdown
## 実装計画
- 変更するファイル / 追加するモジュール:
- DB変更(migration)の有無:
- テスト方針(何をどの粒度で):
- 仕様上の判断が必要な点(あれば):
```

## 4. ブランチを切って実装する

- `git switch -c feature/<issue番号>-<slug>` (例: `feature/11-availability-api`)。
- 命名はすべてdocs準拠。エンドポイントは `docs/api.md`、カラムは `docs/database.md` と一致させる。
- 空き枠・予約系のissueでは先に `reservation-domain` スキルを読む。

## 5. テストを書く

- 受入条件を1つずつテストケースに翻訳する。受入条件に「テストで検証」とあるものはテスト必須。
- 認可(ロール別403/401)と境界値(24時間前ちょうど、営業終了ちょうど)を含める。

## 6. 仕様適合チェック

- `spec-compliance` スキルの手順でdocsと突合し、チェックリストを出力する。

## 7. PRを作成する

- `.github/PULL_REQUEST_TEMPLATE.md` の全セクションを埋める。`Closes #<番号>` を必ず入れる。
- spec-complianceの結果チェックリストをPRに貼る。
- セルフレビュー(`docs/development-flow.md` の観点リスト)をしてからマージする。

## AIとしての振る舞い

- コード生成後は、受講者がレビューすべきポイント(トランザクション境界、境界値、認可)を箇条書きで提示する。
- 受講者が内容を理解しないままマージしようとしていたら、要点の説明を挟む。レビュー責任は受講者にある。
