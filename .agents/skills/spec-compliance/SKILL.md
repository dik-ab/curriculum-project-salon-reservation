# Spec Compliance — 仕様適合チェック

PRを作る前に、実装とdocsを機械的に突合します。結果はチェックリストとして出力し、PRの「spec-complianceチェック結果」セクションに貼ります。

## 手順

1. 今回のdiffを確認する(`git diff main...HEAD`)。
2. diffに含まれる要素ごとに、対応するdocsのセクションを開いて突合する。
3. 下のチェックリストを埋めて出力する。該当しない項目は「N/A」とする。
4. 不一致が1つでもあれば、**PRを作る前に**修正するか、仕様側の問題ならissueとして起票する。

## 突合の対象

| diffに含まれるもの | 突合先 |
|---|---|
| migration / スキーマ定義 | `docs/database.md` のテーブル定義表(カラム名、型、NULL、デフォルト、制約、インデックス) |
| APIエンドポイント | `docs/api.md` の一覧表(メソッド、パス、認証、ロール)とJSON例(キー名、ネスト構造) |
| エラーハンドリング | `docs/api.md` のエラーコード一覧(`code` の値とHTTPステータスの組) |
| ビジネスロジック | `docs/requirements.md` の該当機能要件(F-x)と用語集の定義 |
| 予約・空き枠ロジック | `reservation-domain` スキルのアルゴリズム仕様 |
| 画面 | `docs/screens.md` の構成要素と遷移(画面ID) |

## 出力フォーマット

```markdown
## spec-complianceチェック結果

### DB
- [ ] テーブル名・カラム名・型がdocs/database.mdと一致(対象: <テーブル名>)
- [ ] UNIQUE・CHECK・FK・インデックスが定義どおり
- [ ] status等の文字列値が仕様の値と完全一致

### API
- [ ] メソッド・パスがdocs/api.mdの一覧表と一致(対象: <パス>)
- [ ] 認証・ロール制御が一覧表どおり(未ログイン401 / ロール不足403をテストで確認)
- [ ] レスポンスJSONのキー名・構造がJSON例と一致(snake_case)
- [ ] エラーレスポンスが {"message", "code"} 形式で、codeとHTTPステータスが一覧と一致

### ビジネスルール
- [ ] 該当する機能要件(F-x)の記述をすべて満たす
- [ ] 境界値の挙動が仕様どおり(例: 24時間前ちょうど=可、グリッド外開始時刻=422)

### テスト
- [ ] 受入条件がテストまたは記録された手動確認でカバーされている
- [ ] 「テストで検証」指定の項目に自動テストがある

### 結果
- 不一致: なし / あり(修正済み) / あり(issue #N として起票)
```

## 注意

- 「動くからOK」ではなく「仕様と一致しているからOK」で判定します。仕様より便利な挙動(例: 期限後もキャンセルできる)は**バグ**です。
- チェックを通すための形骸化した「全部✓」を出力しない。実際にファイルを開いて確認した項目だけ✓にする。
