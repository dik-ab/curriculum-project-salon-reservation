# 開発の進め方 — Lumina Reserve

このプロジェクトはissue駆動 + AI駆動(Claude Code / Codex)で進めます。実務のチーム開発と同じ型で、1人でも回せるワークフローを身につけることが目的です。

## 全体の流れ

1. このリポジトリを「Use this template」で自分のリポジトリにする(README参照)。
2. `scripts/setup_issues.sh` でlabel・milestone・issueを自分のリポジトリに複製する(`.claude/skills/setup-github-project` スキル参照)。
3. **M1 → M2 → M3 → M4 の順に、マイルストーン内はissue番号順に**進める。issueは前のissueの成果を前提に書かれているため、原則として順番を守ってください。
4. マイルストーンが終わるごとに、動くデモを自分で触って確認する(M1: ログインできる / M2: 空き枠が出る / M3: 予約が一周する / M4: 運用機能まで動く)。
5. advancedラベルのissueはM4完了後の任意課題です。

## 1 issue = 1 branch = 1 PR

- issueごとにmainからブランチを切ります。ブランチ名は `feature/<issue番号>-<slug>` です(例: issue #11「M2-04: 空き枠計算APIを実装する」→ `feature/11-availability-api`)。
- PRは必ずPRテンプレート(`.github/PULL_REQUEST_TEMPLATE.md`)に沿って書き、本文に `Closes #<issue番号>` を入れます(マージでissueが自動クローズされます)。
- レビュアーがいないため、セルフマージで構いません。ただし後述のセルフレビューを済ませてからマージしてください。
- mainには直接コミットしません(唯一の例外: リポジトリ初期化時のスキャフォールド)。

## issueへの取り組み方(AI駆動の型)

各issueは `.claude/skills/issue-workflow` スキルの手順で進めます。要点は次の通りです。

1. **issueを読む**: タスクと受入条件を理解する。
2. **仕様を特定する**: issueの「仕様参照」からdocsの該当セクションを読む。仕様に書いていないことを実装で勝手に決めない(小さな判断はPRに記録、大きな判断はissueコメントに書いてから進める)。
3. **計画を書く**: 実装計画(変更するファイル、テスト方針)をissueコメントに書く。AIに計画を立てさせた場合も、自分で読んで妥当性を判断してからコメントする。
4. **実装する**: AIに実装を依頼する場合は、仕様参照先のdocsを読ませた上で依頼する。生成されたコードは必ず自分で読み、理解できないコードはそのまま採用しない(説明させる、簡潔に書き直させる)。
5. **テストを書く**: 受入条件をテストコードに翻訳する。issueに「テストで検証」とある項目はテスト必須。
6. **仕様適合チェック**: `.claude/skills/spec-compliance` スキルでdocsとの突合を行い、結果をPRに貼る。
7. **PR作成 → セルフレビュー → マージ**。

> AIは「速い同僚」であって「責任者」ではありません。マージした瞬間にそのコードの責任はあなたに移ります。AIの出力を検証せずマージする癖がつくと、実務で最も信頼を失うタイプの事故を起こします。

## Definition of Done(全issue共通)

以下をすべて満たしたらDoneです。

- [ ] issueのタスクと受入条件をすべて満たしている
- [ ] 自動テストを追加し、既存テストも含めて全件パスしている
- [ ] spec-complianceスキルで仕様適合チェックを実施し、結果をPRに記載した
- [ ] セルフレビュー(下記観点)を実施した
- [ ] PRテンプレートの全セクションを埋め、`Closes #N` を記載した

## セルフレビューの観点

レビュアーがいない分、マージ前に自分のPRのdiffを**他人のPRのつもりで**GitHub上で読みます。最低限、次を確認してください。

**仕様との一致**
- エンドポイントのパス・メソッド・レスポンスのキー名がdocs/api.mdと一字一句一致しているか
- テーブル・カラムの名前と型がdocs/database.mdと一致しているか
- エラー時の `code` とHTTPステータスが仕様のエラーコード一覧と一致しているか

**正しさ**
- 境界値: 「24時間前ちょうど」「営業終了ちょうどに終わる予約」「0件のとき」の挙動をテストしたか
- 認可: そのAPIを別ロール・未ログイン・他人のリソースIDで叩いたテストがあるか
- トランザクション境界: 複数テーブルへの書き込みが1トランザクションに収まっているか

**品質**
- 消し忘れたデバッグ出力・コメントアウトされたコード・未使用importがないか
- 秘密情報(パスワード、トークン)をログに出力していないか、コミットに含めていないか
- AIが生成した不要な抽象化(使われないオプション引数、1箇所からしか呼ばれない汎用ヘルパー)を削ったか

## PRテンプレート

`.github/PULL_REQUEST_TEMPLATE.md` を使います。セクション構成は次の通りです。

| セクション | 書くこと |
|---|---|
| 概要 | 何をなぜ変えたか(1〜3文) |
| 対応issue | `Closes #N` |
| 仕様参照 | 準拠したdocsのセクションへのリンク |
| 変更内容 | 主な変更点の箇条書き |
| テスト | 追加・変更したテストと、手元での実行結果 |
| spec-complianceチェック結果 | スキル実行結果のチェックリストを貼る |

## コミットの粒度

- 1コミット=1つの論理的な変更(「migration追加」「API実装」「テスト追加」のような単位)を推奨します。
- コミットメッセージは `feat: 空き枠計算APIを実装 (#12)` のようにprefix(feat/fix/test/chore/docs)+日本語要約で構いません。

## 困ったとき

- 仕様の解釈に迷う → docs/requirements.mdを正とし、それでも曖昧ならissueコメントに「こう解釈して進める」と書いて進める(実務での「確認して進める」の練習です)。
- 実装が大きくなりすぎる → issueのタスク分割を見直す。1PRの目安は変更500行以内(テスト除く)。超えるならPRを分割する。
- M2-04(空き枠計算)とM3-01(二重予約防止)はこのプロジェクトの山場です。`.claude/skills/reservation-domain` スキルを必ず読んでから着手してください。
