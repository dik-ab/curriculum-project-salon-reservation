# 要件定義書 — Lumina Reserve

美容室「Lumière(リュミエール)」の予約管理システム「Lumina Reserve」の要件定義書です。実装スタックに依存しない仕様をここで固定します。実装中に迷ったときは、必ずこのドキュメントに戻って確認してください。

関連ドキュメント: [DB設計書](./database.md) / [API設計書](./api.md) / [画面定義書](./screens.md) / [インフラ設計書](./infra.md) / [開発の進め方](./development-flow.md)

## 1. 背景とストーリー

株式会社Lumina(ルミナ)は、都内で美容室「Lumière」1店舗を運営するビューティーテック企業です。現在の予約管理は電話と紙台帳で行っており、次の課題を抱えています。

- 営業中は電話対応でスタイリストの手が止まり、施術が中断される
- 紙台帳の転記ミスによるダブルブッキングが月に数回発生している
- 顧客ごとの施術履歴(カルテ)が個人のメモに散在し、担当者が変わると引き継げない
- 無断キャンセル(no show)の傾向を把握できず、対策が打てない

あなたはLuminaに入社した1人目のエンジニアとして、この予約業務をWebシステム化します。まずは1店舗で確実に動くMVPを作り、将来の複数店舗展開に備えます。

### 店舗の基本情報

| 項目 | 内容 |
|---|---|
| 店舗名 | Lumière(リュミエール) |
| 営業時間 | 火〜金 10:00〜19:00、土日 09:00〜18:00 |
| 定休日 | 月曜日 |
| スタイリスト | 3名(店長を除く) |
| メニュー例 | カット60分 / カラー90分 / パーマ120分 / トリートメント30分 / ヘッドスパ30分 |

営業時間と定休日はシステム上で変更可能にします(上記は初期値)。

## 2. アクター(ロール)

| ロール | 説明 | 主な操作 |
|---|---|---|
| `customer` | 来店客。メールアドレスで会員登録して予約する | 空き枠検索、予約作成・変更・キャンセル、マイページ |
| `staff` | スタイリスト。自分のシフトと店舗の予約状況を見る | 予約ボード閲覧、電話予約の登録、来店・no show処理、カルテ記録、自分のシフト管理 |
| `admin` | 店長。店舗運営のすべてを管理する | staffの全操作+メニュー管理、スタッフ管理、営業時間設定、招待発行、売上サマリ |

- ロールは `users.role` カラムで管理し、1ユーザー1ロールです。
- adminはstaffができる操作をすべて実行できます(上位互換)。
- staff・adminのアカウントは一般公開のサインアップでは作成できず、adminが発行する招待トークン経由でのみ登録できます。
- 最初のadminアカウントはシード(初期データ投入)で作成します。招待の発行者が存在しないと運用を開始できないためです。

## 3. 用語集

| 用語 | 英語表記 | 定義 |
|---|---|---|
| 空き枠 | availability slot | あるスタッフが指定メニュー(合計所要時間)の施術を開始できる30分刻みの開始時刻。営業時間とスタッフの勤務時間の共通部分から、既存予約と重なる時刻を除いたもの |
| 所要時間 | total duration | 予約で選択したメニューの `duration_min` の合計(分)。予約の終了時刻は「開始時刻+所要時間」 |
| 指名 | staff nomination | 顧客が施術スタッフを指定して予約すること。指名なしの場合は対応可能な全スタッフから空き枠を横断検索する |
| シフトパターン | shift pattern | スタッフの週次の勤務予定(曜日ごとの開始・終了時刻)。曜日に行がなければその曜日は勤務なし |
| シフト例外 | shift exception | 特定の日付に限りシフトパターンを上書きする設定。`off`(終日休み)と `custom`(その日だけの勤務時間)の2種類 |
| 営業時間 | business hours | 店舗として顧客を受け入れる時間帯。曜日ごとに設定し、定休日は `is_closed = true` |
| 二重予約 | double booking | 同一スタッフの予約時間帯が重複して確定されてしまう状態。本システムで絶対に発生させてはならない |
| 無断キャンセル | no show | 予約時刻に来店せず、連絡もなかった予約。staff・adminが手動でステータスを `no_show` に変更する |
| 電話予約 | phone reservation | 電話で受けた予約をstaff・adminが予約ボードから手動登録すること。`reservations.created_by = 'admin'` で記録する |
| カルテ | customer note | 施術内容・薬剤・会話メモなど、顧客ごとにstaff・adminが記録する施術メモ |
| 招待トークン | invitation token | adminが発行する、staff・admin登録用の使い捨てトークン(有効期限72時間) |

## 4. 機能要件

### 4.1 Must(MVPに必須)

#### F-1. 認証

- customerはメールアドレス・名前・パスワードでサインアップできます。サインアップ直後は仮登録で、確認メールのリンク(トークン付きURL)を開くと本登録になります。メール確認が済むまでログインできません。
- ログインはメールアドレス+パスワードです。認証状態はSNSカリキュラムと同じ **HttpOnly Cookie + JWT** 方式で保持します(詳細は [API設計書 - 認証方式](./api.md#認証方式))。
- パスワードは8文字以上、bcrypt(cost 10以上)または同等以上の方式でハッシュ化して保存します。平文保存は禁止です。
- staff・adminは招待制です。adminがメールアドレスとロールを指定して招待を発行すると、招待メールが届き、リンク先で名前とパスワードを入力して登録します。招待トークンの有効期限は発行から **72時間**、1回使うと無効になります。
- 招待経由で登録したユーザーはメール確認済みとして扱います(招待メールが届いた時点でメール到達を確認できているため)。

#### F-2. メニュー管理(admin)

- adminはメニューを作成・編集できます。項目: 名前、説明、所要時間(分)、価格(税込・円)、有効/無効、並び順。
- 所要時間は1分単位で登録できますが、運用上は30分単位を推奨します(空き枠の開始時刻は30分刻みのため)。
- 無効(`active = false`)にしたメニューは、顧客向けメニュー一覧と空き枠検索の対象から外れます。既存予約には影響しません。
- 予約実績のあるメニューは削除できません(無効化で対応します)。予約実績のないメニューのみ削除できます。
- 顧客向けメニュー一覧は `sort_order` 昇順で表示します。

#### F-3. スタッフ管理(admin)

- adminはスタッフのプロフィール(表示名、自己紹介、画像URL、有効/無効)を編集できます。
- adminはスタッフごとに対応可能メニューを紐付けます。スタッフは紐付いていないメニューの予約を受けられません。
- 無効(`active = false`)のスタッフは指名一覧と空き枠検索の対象から外れます。既存予約には影響しません。
- スタッフのプロフィール行は招待受諾時に自動作成されます(初期値は `active = false`。adminが対応メニューを設定してから有効化する運用)。

#### F-4. シフト管理(admin / staff本人)

- スタッフごとに週次シフトパターン(曜日×開始・終了時刻)を設定します。1スタッフ1曜日につき1行までで、行がない曜日は勤務なしです。
- 特定日の例外(シフト例外)を登録できます。`off` はその日終日休み、`custom` はその日だけの勤務時間の上書きです。1スタッフ1日付につき1件までです。
- シフトはadminが全スタッフ分、staffは自分の分のみ編集できます。
- adminは店舗の営業時間(曜日ごとの開店・閉店時刻、定休日)を設定できます。
- シフトが営業時間の外にはみ出して登録されていても、空き枠計算では営業時間との共通部分だけが使われます(登録自体はエラーにしません。早出・残業の準備時間などを許容するためです)。

#### F-5. 空き枠検索(customer、未ログインでも可)

- 入力: 日付(1日分)、メニュー(1つ以上、複数選択可)、スタッフ(指名は任意)。
- 出力: 開始時刻が30分刻み(毎時00分・30分開始)の空き枠一覧。各枠にはその時刻に対応可能なスタッフが含まれます。
- 空き枠の計算ルール(正確な定義。実装は [reservation-domainスキル](../.claude/skills/reservation-domain/SKILL.md) を参照):
  1. 所要時間 = 選択したメニューの所要時間の合計。
  2. 対象スタッフ = 指名があればその1名、なければ「選択メニューすべてに対応可能」かつ有効なスタッフ全員。
  3. スタッフごとの勤務ウィンドウ = 営業時間 ∩ その日のシフト(シフト例外がある日は例外が優先。`off` なら勤務なし)。定休日・パターン未登録曜日は勤務なし。
  4. 勤務ウィンドウ内で、開始時刻が毎時00分または30分で、かつ「開始時刻+所要時間」がウィンドウ終了時刻以内に収まる時刻を候補にします。
  5. 候補から、そのスタッフの既存予約(ステータス `confirmed`)と時間帯が1分でも重なる時刻を除外します。
  6. 検索日が当日の場合、開始時刻が「現在時刻+60分」より前の枠は返しません(準備時間の確保のため)。
- 検索は1リクエストのAPI呼び出しで完結させます(スタッフ×枠ごとにクエリを発行するN+1実装は不可)。

#### F-6. 予約(customer)

- 顧客は空き枠検索の結果から枠を選んで予約を確定します。予約確定にはログインが必要です。
- 予約の属性: 担当スタッフ、メニュー(複数可)、開始時刻、終了時刻(=開始+所要時間合計)、合計金額(=選択メニュー価格の合計)、顧客からの要望メモ(任意)。
- 価格と所要時間は予約時点の値を `reservation_menus` にスナップショット保存します(後からメニューの価格が変わっても過去の予約金額は変わりません)。
- **二重予約防止**: 予約作成はトランザクション内で行い、`SELECT ... FOR UPDATE` によるロックで同一スタッフへの同時リクエストを直列化してから重複チェックを行います。同一スタッフの `confirmed` 予約と時間帯が重なる予約は絶対に作成できません。同時リクエストの競合は自動テストで検証します(詳細は [DB設計書 - 二重予約の防止](./database.md#二重予約の防止) と reservation-domainスキル)。
- 予約の**変更・キャンセルは開始時刻の24時間前まで**です。「開始時刻の24時間前ちょうど」までは可能、それを1秒でも過ぎたら不可です(例: 7/15 14:00開始の予約は 7/14 14:00:00 まで変更・キャンセル可能)。変更の場合、期限判定は**変更前の開始時刻**を基準に行い、変更後の枠は空き枠のルール(F-5。当日+60分ルールを含む)を満たす必要があります。期限を過ぎた場合は店舗へ電話し、staff・adminが予約ボードから操作します(staff・adminの操作に24時間ルールは適用されません)。
- キャンセルは物理削除せず、ステータスを `cancelled` に変更して `cancelled_at` と任意の理由を記録します。
- 予約ステータスは `confirmed`(確定) / `cancelled`(キャンセル) / `completed`(来店・施術完了) / `no_show`(無断キャンセル)の4値です。新規作成時は必ず `confirmed` で、遷移は `confirmed` からの一方向のみです(遷移図は [DB設計書](./database.md#reservationsのステータス遷移))。
- `completed` と `no_show` への変更はstaff・adminのみが行えます。customerが行えるのは `cancelled` への変更(=キャンセル)だけです。

#### F-7. マイページ(customer)

- 今後の予約一覧(`confirmed` かつ開始時刻が現在以降)を開始時刻の昇順で表示します。
- 過去の予約一覧(上記以外。`completed` / `cancelled` / `no_show` / 開始済みの `confirmed`)を開始時刻の降順で表示します。
- 予約詳細(メニュー内訳、担当スタッフ、金額)を確認し、期限内であれば変更・キャンセルできます。
- プロフィール(名前、メールアドレス、パスワード)を編集できます。メールアドレス変更時の再確認メールは発展課題とし、MVPでは名前とパスワードの変更のみ必須とします。

#### F-8. 予約ボード(admin / staff)

- 日付を指定して、その日の全スタッフの予約をタイムライン形式(縦軸=時刻、横軸=スタッフ)で一覧できます。
- 電話予約を手動登録できます。既存顧客を検索して選ぶか、名前+メールアドレスでその場で簡易顧客登録して予約を作成します(簡易登録した顧客はランダムパスワードのcustomerとして作成し、ログインは本人がパスワード再設定するまで不要です)。
- 手動登録でも空き枠のルール(F-5)と二重予約防止(F-6)は同じように適用されます。ただし「当日+60分」ルールは適用しません(店頭・電話での直前受付を許容するため)。
- 予約に対して来店処理(`completed`)、無断キャンセル処理(`no_show`)、店舗都合キャンセル(`cancelled`)を実行できます。`completed` / `no_show` にできるのは開始時刻を過ぎた予約のみです。

#### F-9. 顧客管理(admin / staff)

- 顧客一覧を名前・メールアドレスの部分一致で検索できます。
- 顧客詳細で、来店履歴(予約履歴)とカルテを時系列で閲覧できます。
- カルテ(施術メモ)を記録できます。項目: 本文、施術日、記録者。カルテは顧客本人には公開しません(社内メモ)。

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| 二重予約防止 | 同一スタッフ・重複時間帯への同時予約リクエストが競合しても、確定されるのは必ず1件のみ。並行リクエストを発行する自動テストで検証する |
| 性能 | 空き枠検索APIは、スタッフ数×枠数に比例したクエリを発行しない(N+1回避)。1リクエストで必要なデータをまとめて取得して計算する |
| 個人情報保護 | パスワードはハッシュのみ保存。メール確認必須。カルテ・顧客一覧はstaff・adminのみアクセス可能で、ロール別アクセス制御を自動テストで担保する |
| タイムゾーン | 日時はUTCで保存(`TIMESTAMPTZ`)し、表示・入力はJST(Asia/Tokyo)。営業時間・シフトの `TIME` 型はJSTの壁時計時刻として扱う |
| テスト | 各マイルストーンの機能に対して自動テストを書く。特に空き枠計算と二重予約防止はテスト必須(該当issueの受入条件に含む) |
| メール | ローカル開発ではMailHogで受信確認する。本番想定はAmazon SES([インフラ設計書](./infra.md)) |

## 6. ユースケース

主要な3本のフローを記述します。画面IDは [画面定義書](./screens.md) を参照してください。

### UC-1: 顧客が予約する

| 項目 | 内容 |
|---|---|
| アクター | customer |
| 事前条件 | メニューとスタッフ・シフトが登録済みで、営業日に空き枠がある |
| 事後条件 | `confirmed` の予約が1件作成され、マイページに表示される |

基本フロー:

1. 顧客がトップページ(C-01)からメニュー選択(C-05)へ進み、メニューを1つ以上選ぶ(例: カット+トリートメント=合計90分)。
2. 日時・スタッフ選択(C-06)で日付を選ぶ。指名したい場合はスタッフも選ぶ。システムが空き枠API(`GET /api/availability`)を呼び、30分刻みの空き枠を表示する。
3. 顧客が枠を選ぶと確認画面(C-07)へ進む。未ログインの場合はログイン画面(C-04)を挟み、ログイン後に確認画面へ戻る。
4. 確認画面でメニュー内訳・担当スタッフ・開始/終了時刻・合計金額を確認し、要望メモ(任意)を入力して確定する。
5. システムが `POST /api/reservations` をトランザクション内で処理し、予約を作成する。完了画面(C-08)を表示する。

代替フロー:

- 3a. 選んだ枠が確定直前に他の顧客に取られた場合、APIは `409 RESERVATION_CONFLICT` を返す。画面はその旨を表示し、日時選択(C-06)へ戻して最新の空き枠を再取得する。
- 4a. 確認画面表示中に日付をまたいで「当日+60分」ルールに抵触した場合、APIは `422 SLOT_UNAVAILABLE` を返す。3aと同様に再検索へ誘導する。

### UC-2: 顧客がキャンセルする

| 項目 | 内容 |
|---|---|
| アクター | customer |
| 事前条件 | 自分の `confirmed` 予約が存在する |
| 事後条件 | 予約のステータスが `cancelled` になり、空き枠が解放される |

基本フロー:

1. 顧客がマイページ(C-09)の今後の予約から対象を選び、予約詳細(C-10)を開く。
2. 現在時刻が「開始時刻の24時間前」以前であれば、キャンセルボタンが有効になっている。
3. 顧客がキャンセル理由(任意)を入力して確定する。システムは `DELETE /api/reservations/:id` を呼び、ステータスを `cancelled` に変更して `cancelled_at` を記録する。
4. 直後から、その時間帯は空き枠検索で再び表示される。

代替フロー:

- 2a. 期限を過ぎている場合、キャンセルボタンは無効で「開始24時間前を過ぎたため、店舗(電話)へご連絡ください」と表示する。APIも `422 CANCEL_DEADLINE_PASSED` で拒否する(画面制御だけに頼らない)。
- 3a. 既に `cancelled` 済みの予約に対するリクエストは `422 INVALID_STATUS_TRANSITION` で拒否する。

### UC-3: 店長が電話予約を登録する

| 項目 | 内容 |
|---|---|
| アクター | admin(staffも同じ操作が可能) |
| 事前条件 | 営業時間・シフト・メニューが登録済み |
| 事後条件 | `created_by = 'admin'` の `confirmed` 予約が作成され、予約ボードに表示される |

基本フロー:

1. 店長が電話で日時・メニュー・指名の希望を聞き取り、予約ボード(A-02)から手動登録フォーム(A-03)を開く。
2. メニューとスタッフ、日付を選ぶと空き枠が表示される(customer向けと同じ計算。ただし「当日+60分」ルールは適用しない)。
3. 顧客欄で氏名またはメールアドレスを検索する。既存顧客ならそれを選択する。
4. 新規顧客なら名前とメールアドレスを入力し、簡易顧客登録と同時に予約を作成する(`POST /api/admin/reservations`)。
5. 予約ボードに新しい予約が表示され、電話予約登録(`created_by = 'admin'`)であることが識別できる。

代替フロー:

- 4a. 入力したメールアドレスが既存顧客と重複する場合、APIは `409 EMAIL_TAKEN` を返す。画面は既存顧客として検索・選択するよう促す。
- 5a. 登録直前に同じ枠が埋まった場合は `409 RESERVATION_CONFLICT`。最新の空き枠を再表示する。

## 7. スコープ外(このプロジェクトでは作らない)

- 複数店舗の同時運用(A-2は設計のみ)
- 決済(A-1のStripeテスト決済を除く)、ポイント・クーポン
- パスワード再設定メール(実装は任意。簡易顧客登録者のログインには必要になるため、余力があれば追加を推奨)
- SMS・LINEなどメール以外の通知チャネル
- スタッフの給与・勤怠管理(シフトはあくまで予約受付のための勤務予定)
- ネイティブアプリ(Webのみ)
